Deployment: Access & Reproduction

Acme Logistics inbound carrier sales — FastAPI backend & operations dashboard. Replace <your-railway-host> with your live hostname (no https:// in the placeholder — the full URL is shown in each row).

1. How to access the deployment

Resource How to access
Operations dashboard Open https://<your-railway-host>/dashboard in a browser. The server injects the API key into the page. Do not open a static HTML file from disk.
API root https://<your-railway-host>/ redirects to the dashboard.
Health check (no API key) GET https://<your-railway-host>/health — returns JSON: status, timestamp, version.
All other API routes Send header X-API-Key: <your API_KEY>. Examples: GET /api/metrics, POST /api/loads/search, POST /api/calls/log.

HTTPS: Railway terminates TLS at the edge; use https:// for all production URLs.

Credentials: Set API_KEY in Railway (or .env locally). Anyone who can load /dashboard can see the injected key in browser tools — treat the dashboard URL as an admin surface and use a strong, rotatable key.

HappyRobot: Configure tool and webhook base URLs in the HappyRobot workflow to https://<your-railway-host> so carrier verification, load search, negotiation, and call logging reach this deployment.

2. How to reproduce the deployment

This repository does not use Terraform. Use one of the following manual paths.

Option A — Railway (production-like)

  1. Prerequisites: GitHub account, Railway account, access to the repository.
  2. Create the service: Railway → New Project → Deploy from GitHub → select the repo. Railway uses railway.json and the Dockerfile to build.
  3. Environment variables (Railway → your service → Variables):
Variable Required Purpose
API_KEY Yes Authenticates API requests; value is injected into /dashboard.
PORT No Railway sets this automatically; the app listens on ${PORT:-8000}.
FMCSA_WEBKEY No Enables the primary FMCSA mobile API; optional fallback applies if unset.
DB_PATH No Default data/carrier_sales.db. Point to a volume path if you attach persistent storage.
  1. Persistence (optional): Mount a volume at /app/data (or set DB_PATH to match) so SQLite survives redeploys.
  2. Deploy: Push to the connected branch; Railway rebuilds and deploys. Open the generated public URL + /dashboard.

Start command (from railway.json): uvicorn app.main:app --host 0.0.0.0 --port ${PORT:-8000}, with automatic restarts on failure (up to 10 retries).

Option B — Docker Compose (local or any Docker host)

From the repository root:

cp .env.example .env
# Edit .env: set API_KEY and optionally FMCSA_WEBKEY
mkdir -p data
docker compose up --build

Then open http://localhost:8000/dashboard. The compose file mounts ./data so the SQLite database persists on the host.

Option C — Local Python (development)

python -m venv venv
source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
mkdir -p data
uvicorn app.main:app --reload --port 8000

On Windows, use venv\Scripts\activate instead of source venv/bin/activate. Use http://localhost:8000/dashboard and send X-API-Key on API requests.

Optional — Web voice test client (QA only)

For browser test calls: run the server/ token service (port 3001) and client/ (Vite) as described in the repository README. Production traffic uses the HappyRobot web-call trigger; this stack is for local testing only.

3. Container summary

  • Base image: python:3.12-slim; dependencies from requirements.txt; application code in /app.
  • Runtime: Single Uvicorn process (appropriate for demo scale; horizontal scaling would add replicas behind the platform load balancer).
